# 全栈开发通用原则

## Context
从一次完整的全栈开发工作中沉淀的经验，覆盖 Bugfix、新功能、性能优化、架构设计、协作沟通、自动化。

## Core Principles

### 1. 需求澄清前置
**When**: 项目开始前
**Do**:
- 用10分钟明确用户场景和期望结果
- 识别模糊点并文档化
- 定义验收标准（Definition of Done）

### 2. 技术方案先评审
**When**: 架构决策时
**Do**:
- 对比2-3个技术选项
- 记录决策理由（ADR格式）
- 团队评审后再动手

### 3. 小步快跑，频繁验证
**When**: 开发过程中
**Do**:
- 每2-4小时完成一个可验证的功能点
- 及时运行测试（而非最后集中测试）
- 保持功能可用状态

### 4. 引入新工具要渐进
**When**: 需要引入新技术/工具时
**Do**:
- 先在非核心功能试点
- 评估学习成本和迁移成本
- 制定回滚方案

### 5. 重构与功能开发分离
**When**: 处理代码质量时
**Do**:
- 新功能遵循新规范
- 旧代码重构单独成任务
- Boy Scout Rule：每次提交让代码更干净

### 6. 性能优化基于数据
**When**: 性能优化场景
**Do**:
- 先测量，再优化
- 建立性能基线
- 优化后验证改善效果

### 7. 信息透明，减少信息孤岛
**When**: 团队协作时
**Do**:
- 关键决策记录在共享文档
- 使用看板可视化工作状态
- 异步沟通优先

### 8. 文档即代码
**When**: 需要知识传递时
**Do**:
- README即入门指南
- 架构文档与代码同步
- API文档自动生成

### 9. 开发环境即代码
**When**: 环境配置时
**Do**:
- 使用Docker统一开发环境
- 环境配置版本控制
- 一键启动脚本

### 10. 自动化重复工作
**When**: 有重复性任务时
**Do**:
- CI/CD自动化构建部署
- 代码格式化工具
- Git hooks自动化检查

## Decision Tree

```
遇到问题？
├── 需求不清 → 原型确认 + 文档记录
├── 技术决策 → 对比分析 + 专家评审
├── 沟通不畅 → 异步文档 + 可视化看板
├── 代码质量 → 代码审查 + 自动化检查
├── 资源不足 → MVP优先 + 迭代交付
└── 环境问题 → 容器化 + 环境即代码
```

## Checklist Template

```
□ 需求是否澄清完毕？
□ 技术方案是否已评审？
□ 开发环境是否准备就绪？
□ 是否有可复用的代码/组件？
□ 性能基线是否已建立？
□ 团队是否了解关键决策？
□ 自动化流程是否就绪？
```
